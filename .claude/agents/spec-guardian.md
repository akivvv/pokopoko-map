3. `D:\pg\AI\ClaudeCode\pokomap-web\docs\SPEC.md` — 機能仕様(旧版ベース、§4〜9の機能は基本踏襲)
4. `D:\pg\AI\ClaudeCode\pokomap-web\docs\HANDOFF.md` — 旧実装のハマりどころ

## 特に監視すべき「確定済みの方針転換」(旧仕様の混入がバグになる)

1. **生成AI連携は完全廃止**。AIプロンプト出力・生成マップ取込が紛れ込んだら即指摘
2. **ピンの kind 種別(pokemon/landmark/item)は廃止**。新モデルは「入れ物」+「中身(ポケモンのみ)」の2概念。入れ物は子入れ物を固定2段まで。residents 最大20匹の上限も撤廃
3. **ピン座標は画像pxではなくドット(グリッド)単位**。pixels と同じ格子で整数比較できること。
   **グリッドサイズは可変**(`maps/<mapId>/meta.grid`、初期46)— 46 のハードコードを見つけたら即指摘(DECISIONS §3)
4. **背景画像は Firebase に載せない**(ローカルのみ)。サーバーが持つのは pins と pixels のみ
5. **同期は onValue(全ノード再送)禁止**。onChildChanged 等の子ノード監視で差分同期
6. **1匹1住処は pokemonLocations 占有インデックス + transaction() で構造的に保証**(クライアント側ソフトチェックのみは不可)
7. **ログは破壊的操作のみ・直近200件で間引き**。paint / savePin は記録しない
8. **認証はハイブリッド(Google推奨+匿名認証、招待コードで部屋参加、メンバーシップ制 Rules)**。
   旧あいことばハッシュ=DBパス方式と、**カスタムトークン方式(REBUILD §2 記載)は廃止済みの旧案**。
   混入したら即指摘(DECISIONS.md §2 が正)
9. **既存データは JSON エクスポート→インポート機能で移行**。DB 直接移行スクリプトは作らない

## 報告形式

- 「仕様適合 / 逸脱 / 仕様側が未定義」の3区分で判定する(DECISIONS.md にも SPEC にも根拠がない場合のみ「未定義」)
- 逸脱は該当ドキュメントの節番号を添えて引用する。その際 `docs/IMPLEMENTATION-NOTES.md` の
  Deviations を必ず参照し、**記録済みの意図的逸脱**(状態と理由を添えて報告)と
  **記録なき逸脱**(記録漏れ or 事故。重点指摘)を区別する
- 未定義の領域に踏み込む実装を見つけたら「設計判断が必要」として報告し、勝手に方針を推定しない
